# 易支付 V1/V2 能力评审与系统边界

> 评审日期：2026-07-18  
> V1 文档：`https://pay.yunvix.com/doc_old.html`  
> V2 文档：`https://pay.yunvix.com/doc/index.html`  
> 结论：新系统默认对接 V2，V1 仅作为迁移兼容；不重复开发易支付已经提供的通道和账务查询能力。

## 1. 核心结论

易支付已经具备完整的上游支付通道能力，包括创建支付、单笔订单查询、订单列表、商户信息、退款、退款查询和关闭订单。V1 还提供结算记录查询，V2 则升级为 RSA 签名、时间戳校验和更完整的支付/退款/关闭订单接口。

因此，本项目不应再建设一套与易支付后台相同的“全量订单中心、结算中心、余额中心、退款引擎或通道管理系统”。本项目应聚焦于易支付没有解决的产品层能力：

- 自定义金额收款页和 API 固定金额收银台；
- 幂等创建、刷新恢复和安全 checkout token；
- 本地业务订单与易支付订单映射；
- 回调验签、实际金额核对和异常状态；
- 开发者应用、签名 API 和 webhook；
- 管理员到账通知及通知重试；
- 本地异常、回调、通知和审计运营台。

## 2. V1 能力清单

### 2.1 支付与回调

| 能力 | 地址 | 说明 |
| --- | --- | --- |
| 页面跳转支付 | `https://pay.yunvix.com/submit.php` | GET/POST，推荐 POST；`type` 不传时进入易支付收银台 |
| API 接口支付 | `https://pay.yunvix.com/mapi.php` | 返回 `payurl`、`qrcode`、`urlscheme` 三者之一 |
| 支付结果通知 | 商户传入的 `notify_url`、`return_url` | GET；成功状态为 `TRADE_SUCCESS`；异步通知需要返回 `success` |
| 签名 | MD5 | 参数名 ASCII 排序，排除 `sign`、`sign_type` 和空值，末尾拼接商户密钥 |

V1 回调明确包含 `money`，可作为验签后的通道支付金额。它不能直接使用，仍需校验 `pid`、签名、订单映射、交易状态和交易号唯一性。

### 2.2 查询与管理

| 能力 | 地址/操作 | 关键结果 |
| --- | --- | --- |
| 查询商户信息 | `api.php?act=query` | 商户状态、余额、结算方式、结算账号、订单统计 |
| 查询结算记录 | `api.php?act=settle` | 结算记录列表 |
| 查询单个订单 | `api.php?act=order` | 易支付单号、商户单号、接口方单号、金额、状态、时间、付款者标识 |
| 批量查询订单 | `api.php?act=orders` | 分页订单列表；`limit` 最大 50 |
| 提交退款 | `api.php?act=refund` | 支持易支付单号或商户单号；后台需开启退款 API |

V1 的问题不是缺少基础能力，而是安全和接口规范较旧：使用 MD5、直接传商户密钥、缺少时间戳、没有退款查询和关闭订单接口。

## 3. V2 能力清单

### 3.1 协议升级

- 数据格式：`application/x-www-form-urlencoded`；
- 返回格式：JSON；
- 签名算法：`SHA256WithRSA`；
- 请求使用商户私钥签名；
- 平台响应和回调使用平台公钥验签；
- 请求和响应包含 10 位秒级 `timestamp`；
- 新系统必须支持回调未来增加扩展字段，不能用固定字段白名单破坏验签。

### 3.2 统一下单

`POST https://pay.yunvix.com/api/pay/create`

核心请求字段：

- `pid`、`method`、`type`、`out_trade_no`；
- `notify_url`、`return_url`；
- `name`、`money`、`clientip`；
- `device`、`param`；
- `timestamp`、`sign`、`sign_type`；
- JSAPI、付款码、小程序、APP 和进件通道所需的可选参数。

核心返回字段：

- `code`、`msg`；
- `trade_no`；
- `pay_type`；
- `pay_info`；
- `timestamp`、`sign`、`sign_type`。

`pay_type` 可能为 `jump`、`html`、`qrcode`、`urlscheme`、`jsapi`、`app`、`scan`、`wxplugin` 或 `wxapp`。本项目首期只需规范化处理网页收款所需的 `jump`、`qrcode` 和必要的 `urlscheme`；其他类型留在适配器中，不泄漏到业务域。

### 3.3 订单查询

`POST https://pay.yunvix.com/api/pay/query`

- `trade_no` 与 `out_trade_no` 必传其一；
- 返回上游订单号、商户订单号、接口订单号、支付方式、支付状态、创建/完成时间、金额、已退款金额、业务参数、付款用户标识和 IP；
- 状态值：`0` 未支付、`1` 已支付、`2` 已退款、`3` 已冻结、`4` 预授权。

该接口用于单笔状态恢复、同步返回后的确认、长时间 pending 补偿和人工刷新，不需要自行重新开发上游订单查询逻辑。

### 3.4 订单列表

`POST https://pay.yunvix.com/api/merchant/orders`

- `offset` 从 0 开始；
- `limit` 最大 50；
- 可选 `status`：`0` 未支付、`1` 已支付；
- `data` 的订单结构参考单笔订单查询。

该接口可直接用于对账、增量同步和上游收款记录查询。不要为了展示全量收款记录而把所有易支付订单永久复制到本地数据库。

### 3.5 支付结果通知

V2 通知明确包含：

- `pid`、`trade_no`、`out_trade_no`、`api_trade_no`；
- `type`、`trade_status`；
- `addtime`、`endtime`；
- `name`、`money`、`param`、`buyer`；
- `timestamp`、`sign`、`sign_type`。

`trade_status` 固定成功值为 `TRADE_SUCCESS`。异步通知持久化成功后返回 `success`。实际到账展示金额可以使用验签后的 `money`，也可由已验签的订单查询结果补偿确认。

### 3.6 退款、退款查询和关闭订单

| 能力 | 地址 | 说明 |
| --- | --- | --- |
| 订单退款 | `POST /api/pay/refund` | 支持商户退款单号 `out_refund_no` 防重复；少数插件不支持部分退款 |
| 退款查询 | `POST /api/pay/refundquery` | 查询平台或商户退款单结果 |
| 关闭订单 | `POST /api/pay/close` | 仅部分支付插件支持；失败时仍需保留本地取消语义 |

退款 API 需要在易支付商户后台开启。我们只需要封装、授权、审计和展示，不需要实现独立退款引擎。

### 3.7 商户能力

V2 文档提供：

- `POST /api/merchant/info`：商户信息；
- `POST /api/merchant/orders`：订单列表；
- 代付发起、代付查询、可用余额查询。

当前产品不做代付，不应因为接口存在而扩大首期范围。

## 4. 能力归属矩阵

| 能力 | 处理方式 | 归属说明 |
| --- | --- | --- |
| 支付通道下单 | 直接复用 | 易支付 V2 `/api/pay/create` |
| 二维码/跳转参数 | 封装复用 | 适配器把 `pay_type/pay_info` 规范化为本地支付动作 |
| 单笔订单状态 | 直接复用 | V2 `/api/pay/query` |
| 上游订单列表 | 直接复用 | V2 `/api/merchant/orders`，用于查询和对账 |
| 商户余额/信息 | 直接复用 | 易支付商户信息接口或易支付后台 |
| 结算记录 | 直接复用 | V1 已提供；优先使用易支付后台，不在本地重建结算账本 |
| 退款 | 封装复用 | V2 refund/refundquery，本地只做权限、幂等和审计 |
| 关闭订单 | 封装复用 | V2 close，按插件能力判断；不支持时本地放弃 |
| 公开自定义金额收款页 | 本地开发 | 易支付不负责本产品 UI、恢复和异常体验 |
| API 固定金额收银台 | 本地开发 | 包含开发者鉴权、金额锁定和 checkout token |
| 本地幂等 | 本地开发 | 防止网络重试生成重复业务订单 |
| 刷新恢复 | 本地开发 | 保存支付动作和本地恢复令牌 |
| 金额核对 | 本地开发 | 对比本地预期金额与验签后的 `money` |
| 异常状态 | 本地开发 | `amount_mismatch`、`paid_after_cancel` 等是本产品语义 |
| 管理员通知 | 本地开发 | 易支付订单记录不能替代主动到账提醒 |
| 开发者 webhook | 本地开发 | 面向本平台 API 开发者，含签名、重试和审计 |
| 本地运营台 | 精简开发 | 只管理本地订单映射、异常、回调、通知、应用和审计 |

## 5. 推荐适配架构

```text
Checkout / Developer API / Operations Console
                  |
          Payment Application Service
                  |
            PaymentGateway Contract
             /                  \
    EasyPayV2Adapter       EasyPayV1Adapter
       default              migration only
             \                  /
          Normalized Payment Result
```

适配器统一输出：

- 上游订单号；
- 商户订单号；
- 规范化支付动作；
- 订单状态；
- 订单金额和退款金额；
- 创建和完成时间；
- 付款者标识；
- 验签结果和上游错误码。

业务层不得直接依赖 `mapi.php` 的 `qrcode/payurl/urlscheme`，也不得直接依赖 V2 的 `pay_type/pay_info`。

## 6. 最小本地数据范围

本地仍必须保存 `payment_orders`，但它不是易支付账本副本。至少需要：

- 本地订单号、来源和开发者应用；
- 外部订单号、幂等键和 checkout token 哈希；
- 预期金额、验签后的实际金额和差额；
- 本地状态和状态时间线；
- 易支付 API 版本、上游订单号和接口订单号；
- 规范化支付动作、二维码内容或跳转链接；
- 创建、过期、取消、支付和最近同步时间；
- 回调指纹、通知和 webhook 关联。

不默认建立：

- 易支付全量订单镜像表；
- 易支付结算记录镜像表；
- 商户余额流水账；
- 支付通道配置和费率系统；
- 独立退款清算引擎。

## 7. 版本选择

### 新项目

直接采用 V2。理由：

- RSA 比共享 MD5 密钥更适合长期开放系统；
- 有时间戳校验；
- 接口路径和返回规范更清晰；
- 支持退款查询和关闭订单；
- 统一下单覆盖网页、二维码、JSAPI、APP、付款码和小程序扩展。

### 现有线上支付

可以短期保留 V1，先完成：

1. 建立统一 `PaymentGateway` 接口；
2. 把当前 `mapi.php` 和 MD5 代码移入 `EasyPayV1Adapter`；
3. 新增 `EasyPayV2Adapter`；
4. 使用配置切换新订单版本；
5. 每个订单保存 `gateway_api_version`；
6. V1 存量订单继续按 V1 验签和查询；
7. 验证稳定后默认切换 V2，最后停用 V1 新单。

不得同时维护两套独立订单状态机。

## 8. 首期范围调整

### 保留开发

- 收款页、移动端支付体验和结果页；
- 本地订单、幂等、恢复和金额校验；
- V2 适配器及 V1 兼容适配器；
- 开发者 API、应用管理和 webhook；
- 管理员到账通知；
- 本地异常、回调、通知、开发者应用和审计页面。

### 删除或降级

- 删除“重新开发完整订单管理后台”的目标；
- 删除“重新开发结算记录、余额和通道管理”的目标；
- 全量收款记录默认使用易支付订单列表或后台；
- 本地订单页只展示由本系统创建或接入的业务订单；
- 财务对账通过易支付订单列表与本地订单做差异核对，不维护第二套上游账本。

## 9. 最终产品定位

本产品不是新的易支付后台，而是建立在易支付之上的轻量支付产品层：

> 易支付负责支付通道、上游订单、余额、结算、退款和关闭订单；本系统负责收银台、开发者接入、可靠恢复、金额可信、异常处理和消息通知。
